# v2.2 Streaming Improvements Design Spec

**Date:** 2026-04-09
**Version:** v2.2
**Priority:** ストリーミング品質の大幅改善 → USB-C / FT / Companion Appの順で段階リリース
**Goal:** 高画質な映像を長時間安定して配信できるストリーミングエンジンの実現

---

## Overview

v2.2はストリーミングエンジンのコア品質を改善するリリース。5つの柱で構成される:

1. レイテンシ最適化（50ms → 35ms目標）
2. 帯域制御の改善（GCCベース + 適応FEC）
3. AV1コーデック対応（30%帯域削減）
4. Foveated Transport（40%帯域削減）
5. 長時間安定性（4時間連続稼働保証）

---

## 1. レイテンシ最適化

### 1.1 フレームパイプライン並列化

**現状:** `pipeline.rs`でフレーム全体のFECエンコード完了後に一括送信。
**改善:** スライス単位でFEC→送信を並列化。デコーダ側で早期デコード開始可能に。

**変更ファイル:**
- `rust/streaming-engine/src/pipeline.rs` — スライスベースFECエンコード + 逐次送信
- `client/app/src/main/cpp/network_receiver.cpp` — スライス単位の受信バッファリング
- `client/app/src/main/cpp/fec_decoder.cpp` — スライス単位FECリカバリ

**設計:**
- フレームをN個のスライスに分割（デフォルト4スライス、MTUサイズベース）
- 各スライスは独立したFECグループとして処理
- スライスごとにFECエンコード完了次第、即座にUDP送信
- クライアント側はスライス0のデータ到着時点でデコーダにキュー投入可能
- FVPヘッダにslice_index（4bit）とslice_count（4bit）を追加

**期待効果:** フレーム送信開始の遅延を 3-5ms → 1-2ms に短縮

### 1.2 NVENCエンコーダ非同期化

**現状:** `nvenc_encoder.cpp`で同期的にエンコード完了を待機。
**改善:** 非同期エンコードAPIを使用し、CPU側フレーム取得とGPUエンコードをオーバーラップ。

**変更ファイル:**
- `driver/src/nvenc_encoder.cpp` — `NV_ENC_PIC_PARAMS`に`completionEvent`設定、`nvEncEncodePicture`非同期呼び出し
- `driver/src/frame_copy.cpp` — 次フレームのGPUコピーをエンコード完了前に開始

**設計:**
- ダブルバッファリング: フレームN+1のD3D11コピー中にフレームNのエンコードを実行
- `completionEvent`（Win32 Event）でエンコード完了を通知
- 最大1フレームの先行読み込み（レイテンシ増加なし、スループット向上）

**期待効果:** エンコードスループット +20%、フレーム落ち率低下

### 1.3 デコード側早期提出

**現状:** `video_decoder.cpp`で全FECグループの復元完了後にデコーダに提出。
**改善:** FEC待ちなしで揃ったデータを即時デコーダに提出。

**変更ファイル:**
- `client/app/src/main/cpp/video_decoder.cpp` — 早期提出ロジック
- `client/app/src/main/cpp/fec_decoder.cpp` — 部分復元状態の管理

**設計:**
- スライス0〜N-1が全て到着: 即時デコーダ提出（FEC不要）
- スライスに欠損あり: FECタイムアウト（5ms）まで待機、タイムアウト後は揃ったスライスで部分デコード
- FEC復元成功時: 次フレームで完全データを提出（エラー伝播を防止）
- 部分デコード時はアーティファクト軽減のためerror concealment有効化

**期待効果:** FEC待ち時間の削減 5-10ms → 0-5ms

### 1.4 適応バッファ制御

**現状:** 固定30msのジッタバッファ。
**改善:** ネットワーク品質に応じて15ms〜50msの範囲で動的調整。

**変更ファイル:**
- `rust/streaming-engine/src/adaptive/bandwidth_estimator.rs` — ジッタ計測の追加
- `client/app/src/main/cpp/network_receiver.cpp` — 動的バッファサイズ
- `config/default.toml` — `[network] jitter_buffer_min_ms`, `jitter_buffer_max_ms`

**設計:**
- パケット間到着時間のジッタ（標準偏差）を計測（直近100パケットの移動窓）
- バッファサイズ = max(min_ms, ジッタ × 3)、上限 max_ms
- ジッタが安定時（σ < 2ms）: バッファ15msまで削減
- ジッタが不安定時（σ > 10ms）: バッファ50msまで拡大
- バッファサイズ変更はスムーズに（1秒あたり最大5msの変化）

**期待効果:** 良好環境で 30ms → 15ms のバッファ削減

### レイテンシ改善合計目標

| 項目 | 現状 | 改善後 |
|------|------|--------|
| パイプライン遅延 | 3-5ms | 1-2ms |
| FEC待ち | 5-10ms | 0-5ms |
| ジッタバッファ | 30ms | 15ms（良好時） |
| **合計** | **50ms** | **35ms（良好時）** |

## 2. 帯域制御の改善

### 2.1 GCC（Google Congestion Control）ベース帯域推定

**現状:** パケットロス率のみに基づく単純なルールベース制御。
**改善:** delay-basedの帯域推定を導入し、ロス発生前に制御。

**新規ファイル:**
- `rust/streaming-engine/src/adaptive/gcc_estimator.rs`

**変更ファイル:**
- `rust/streaming-engine/src/adaptive/bitrate_controller.rs` — GCC統合
- `rust/streaming-engine/src/adaptive/bandwidth_estimator.rs` — delay-based推定の追加

**設計:**
- パケット送信時刻と受信確認時刻の差（one-way delay variation）を計測
- Kalmanフィルタでdelay trendを推定
- delay増加トレンド検出 → 帯域逼迫と判断 → ビットレート削減
- delay安定/減少トレンド → 帯域余裕あり → プロービング許可
- 既存のloss-basedコントローラとの併用: delay-basedとloss-basedの推定値のmin()を採用

**プロトコル変更:**
- HEARTBEATメッセージにパケット受信タイムスタンプ配列を追加（transport-cc相当）
- `common/src/protocol.rs`にTransportFeedbackメッセージ（0x11）を追加

### 2.2 適応FEC冗長度

**現状:** 固定20%冗長度。
**改善:** パケットロス率に応じて5%〜40%で動的調整。

**変更ファイル:**
- `rust/streaming-engine/src/transport/fec.rs` — `AdaptiveFecController`構造体追加
- `config/default.toml` — `fec_redundancy_min`, `fec_redundancy_max`

**設計:**
- ロス率 < 1%: FEC 5%（帯域節約モード）
- ロス率 1-3%: FEC 15%
- ロス率 3-5%: FEC 25%
- ロス率 > 5%: FEC 40%（最大保護モード）
- 変更は1秒ごとに評価、1回の変更は最大±5%
- 節約したFEC帯域はビットレートに回す（全体帯域一定）

### 2.3 バースト耐性

**現状:** 短時間のバーストロスでもビットレートを即座に大幅削減。
**改善:** バーストパターンを検知し、FEC一時引き上げで吸収。

**変更ファイル:**
- `rust/streaming-engine/src/adaptive/bitrate_controller.rs` — バースト検知ロジック

**設計:**
- バースト判定: 100ms以内に5%以上のロスが発生し、その前後はロスなし
- バースト検知時: FEC冗長度を一時的に40%に引き上げ（3秒間）
- バーストが500ms以上継続: バーストではなく持続的な品質劣化と判断 → ビットレート削減
- バースト回復後: 5秒かけてFECを元のレベルに戻す

### 2.4 帯域プロービング

**現状:** 10秒安定で+5%の固定増加。
**改善:** プローブパケットで実際の帯域余裕を測定。

**変更ファイル:**
- `rust/streaming-engine/src/adaptive/gcc_estimator.rs` — プロービングロジック

**設計:**
- 10秒以上安定かつdelay trendが減少傾向の時にプロービング開始
- 短いバースト（50ms間）で現在ビットレートの120%のレートでプローブパケットを送信
- delay増加なし → 帯域余裕あり → ビットレートをプローブレートまで引き上げ
- delay増加あり → 現在が上限に近い → 維持
- プロービング間隔: 最小30秒

## 3. AV1コーデック対応

### 3.1 NVENCエンコーダ AV1追加

**変更ファイル:**
- `driver/src/nvenc_encoder.cpp` — AV1エンコードプロファイル追加
- `driver/src/nvenc_encoder.h` — AV1関連の定数・構造体

**設計:**
- `NV_ENC_CODEC_AV1_GUID`によるAV1エンコーダ初期化
- Profile: Main (8bit)
- Level: 5.1（4K@120fps対応）
- Tuning: Ultra Low Latency
- Rate Control: CBR
- GOP: Intra-only refresh（低レイテンシ優先）
- GPU capability check: `nvEncGetEncodeCaps`でAV1サポートを確認
- 非対応GPU: AV1選択時に自動でH.265にフォールバック、ログ警告

### 3.2 Android MediaCodec AV1デコード

**変更ファイル:**
- `client/app/src/main/cpp/video_decoder.cpp` — AV1デコーダパス
- `client/app/src/main/cpp/nal_validator.cpp` — OBUバリデーション追加

**設計:**
- MIME type: `"video/av01"`
- HWデコーダ優先（`MediaCodecList`でHWデコーダを検索）
- Snapdragon XR2 Gen 2: AV1 HWデコード対応確認済み
- Surface直接出力（既存のzero-copyパスを流用）
- HWデコーダ非対応時: AV1選択を拒否、ネゴシエーション時にH.265を要求

### 3.3 RTPパケタイゼーション（OBU）

**変更ファイル:**
- `rust/streaming-engine/src/transport/rtp.rs` — AV1 RTPパケタイザ追加

**設計:**
- RFC 9601（RTP Payload Format for AV1）準拠
- Aggregation header + OBU elements
- フラグメンテーション対応（MTU 1500B超のOBU）
- 既存のFVPヘッダとの共存（FVPヘッダ → AV1ペイロードヘッダ → OBUデータ）

### 3.4 コーデック自動選択更新

**変更ファイル:**
- `rust/streaming-engine/src/codec_benchmark.rs` — AV1ベンチマーク追加
- `rust/common/src/protocol.rs` — CodecType enumにAV1追加
- `rust/common/src/constants.rs` — AV1関連定数

**設計:**
- ベンチマーク: H.264 → H.265 → AV1 の順で各5秒テスト
- 自動選択ロジック:
  1. GPU AV1対応 AND HMD AV1デコード対応 → AV1
  2. GPU H.265対応 AND HMD H.265デコード対応 → H.265
  3. フォールバック → H.264
- ネットワーク帯域が低い場合（< 30Mbps）: AV1を優先（圧縮効率が高い）
- `STREAM_CONFIG`メッセージのcodecフィールドに`AV1 = 3`を追加

### 3.5 Companion App コーデック選択UI更新

**変更ファイル:**
- `rust/companion-app/src/main.rs` — コーデック選択にAV1追加
- `rust/companion-app/src/config.rs` — AV1設定の読み書き

**設計:**
- Settings タブのCodec選択: Auto / H.264 / H.265 / AV1
- AV1非対応GPU時: AV1オプションをグレーアウト + ツールチップ「RTX 40シリーズ以降が必要」
- ベンチマーク結果にAV1のレイテンシ・画質スコアを追加表示

## 4. Foveated Transport

### 4.1 2ストリーム分割エンコード

**変更ファイル:**
- `driver/src/nvenc_encoder.cpp` — デュアルエンコーダインスタンス管理
- `driver/src/nvenc_encoder.h` — FoveatedEncoder構造体
- `driver/src/frame_copy.cpp` — フォビア/周辺領域のGPUクロップ
- `driver/src/qp_map.h` — ストリーム分割パラメータ計算

**設計:**
- **フォビアストリーム:** 注視点中心に解像度の60%をカバーする矩形をフル解像度でエンコード
- **周辺ストリーム:** フレーム全体を1/2解像度（各軸）でエンコード
- D3D11でフォビア領域をクロップコピー → NVENCインスタンス1
- D3D11でフレーム全体を1/2にダウンスケール → NVENCインスタンス2
- フォビア領域の座標は視線データから計算（前フレームの視線位置 + 予測補正）
- NVENCインスタンスは2つ同時使用（GPUリソース管理が必要）

**プリセット:**

| プリセット | フォビア解像度 | 周辺解像度 | 帯域削減目標 |
|-----------|--------------|-----------|------------|
| subtle | 70% area, full res | 50% res | 25% |
| balanced | 60% area, full res | 40% res | 40% |
| aggressive | 50% area, full res | 25% res | 55% |

### 4.2 RTPストリーム多重化

**変更ファイル:**
- `rust/streaming-engine/src/transport/rtp.rs` — ストリームID対応
- `rust/streaming-engine/src/pipeline.rs` — デュアルストリームパイプライン

**設計:**
- FVPヘッダのreservedビットにstream_id（2bit）を追加:
  - 0: single stream（従来互換）
  - 1: fovea stream
  - 2: periphery stream
- 各ストリームは独立したRTPシーケンス番号を持つ
- FECは各ストリームごとに独立適用
- フォビアストリームの優先度を高く設定（DSCP EF維持、周辺はAF41）

### 4.3 クライアント側合成レンダリング

**変更ファイル:**
- `client/app/src/main/cpp/video_decoder.cpp` — デュアルデコーダ
- `client/app/src/main/cpp/timewarp.cpp` — 合成シェーダ
- 新規: `client/app/src/main/assets/foveated_composite.frag` — 合成フラグメントシェーダ

**設計:**
- MediaCodecインスタンス2つ（fovea + periphery）
- フォビアテクスチャ: GL_TEXTURE_EXTERNAL_OES（zero-copy）
- 周辺テクスチャ: GL_TEXTURE_EXTERNAL_OES（zero-copy）
- 合成シェーダ:
  1. 周辺テクスチャをバイリニアアップスケールでフル解像度に
  2. フォビア領域をオーバーレイ（ソフトエッジブレンド、8pxグラデーション）
  3. timewarp回転補正を合成後に適用
- フォビア領域のスクリーン座標はTCPで受信（エンコーダ側と同期）

### 4.4 視線追跡連動

**変更ファイル:**
- `client/app/src/main/cpp/eye_tracker.cpp` — 視線予測
- `driver/src/nvenc_encoder.cpp` — 視線データ受信、フォビア位置更新

**設計:**
- 視線データ送信頻度: 毎フレーム（TRACKING_DATAメッセージに含まれる既存パス）
- 視線予測: 直近3フレームの視線位置から線形予測（エンコードレイテンシ分を補償）
- サッケード検出: フレーム間の視線移動距離 > 5度 → サッケードと判定
- サッケード中: 一時的にフォビア領域を拡大（画面の80%）するか、single stream にフォールバック
- サッケード終了後: 200msかけて通常のフォビアサイズに縮小

## 5. 長時間安定性

### 5.1 メモリリーク防止

**変更ファイル:**
- `rust/streaming-engine/src/pipeline.rs` — バッファプール上限 + リサイクル
- `rust/streaming-engine/src/transport/fec.rs` — LRU eviction
- 新規: `rust/streaming-engine/src/metrics/memory.rs` — メモリ監視

**設計:**
- バッファプール: 最大256バッファ、超過時は最古のバッファを解放
- FECエンコーダキャッシュ: LRU、最大16エントリ、5分未使用で解放
- メモリ監視: 60秒ごとにヒープ使用量を記録、1時間で50MB以上の増加を検知 → 警告ログ
- `jemalloc`のstats APIでヒープフラグメンテーションも監視

### 5.2 エンコーダリカバリ

**変更ファイル:**
- `driver/src/nvenc_encoder.cpp` — タイムアウト + 自動リセット
- `driver/src/nvenc_encoder.h` — リカバリ状態管理

**設計:**
- エンコードタイムアウト: 100ms（通常は3-5ms）
- タイムアウト検知: `WaitForSingleObject`のタイムアウト
- リカバリ手順:
  1. 現行エンコーダセッションを`nvEncDestroyEncoder`で破棄
  2. 新規セッションを`nvEncOpenEncodeSession`で再作成
  3. クライアントに`IDR_REQUEST`を送信してキーフレーム要求
  4. 次フレームからエンコード再開
- 連続リカバリ（30秒以内に3回）: GPU深刻問題と判断 → エラーログ + ユーザー通知
- リカバリ中のフレーム欠損: クライアント側でlast-frame表示（フリーズ、100-200ms程度）

### 5.3 TCP制御チャネル堅牢化

**変更ファイル:**
- `rust/streaming-engine/src/control/tcp_server.rs` — 再接続ロジック
- `rust/streaming-engine/src/control/tls.rs` — セッション再開

**設計:**
- ハートビート途絶検知: 3回連続欠損（1.5秒）で接続断と判定
- 再接続フロー:
  1. サーバー側はストリーミング状態を5秒間保持（即座に停止しない）
  2. クライアントは自動再接続を試行（1秒間隔、最大5回）
  3. TLS session resumption（session ticket）で再接続時のハンドシェイクを高速化
  4. 再接続成功: ストリーミング継続（PIN再入力不要）
  5. 再接続失敗（5回）: ストリーミング停止 + Companion App通知
- ストリーミング中のTCPウィンドウサイズ: 制御メッセージは小さいのでデフォルト維持

### 5.4 適応パフォーマンス制御（Thermal Governor）

**変更ファイル:**
- 新規: `rust/streaming-engine/src/thermal.rs` — GPU温度監視
- `rust/streaming-engine/src/engine.rs` — thermal governor統合
- `rust/streaming-engine/Cargo.toml` — nvml-wrapper依存追加

**設計:**
- NVML API経由でGPU温度を5秒ごとに取得
- 温度閾値:
  - < 75°C: 通常動作
  - 75-85°C: 警告 → 解像度を90%に下げる
  - 85-90°C: ビットレートを70%に制限 + 解像度80%
  - > 90°C: 緊急 → 最低品質設定にフォールバック
- 温度低下後: 30秒かけて段階的に品質復帰
- Companion Appに温度状態を通知（status.json経由）
- NVML非対応環境: thermal governor無効、ログ警告のみ

### 5.5 ストリーミング統計永続化

**変更ファイル:**
- 新規: `rust/streaming-engine/src/metrics/session_log.rs` — セッション記録
- `rust/streaming-engine/src/engine.rs` — ログ統合
- `rust/companion-app/src/main.rs` — セッション履歴UI

**設計:**
- ログ形式: JSON Lines（`.jsonl`）
- ログ場所: `%APPDATA%/FocusVisionPCVR/sessions/`
- 記録間隔: 10秒ごとに1レコード
- レコード内容:
  ```json
  {
    "ts": "2026-04-09T12:00:00Z",
    "latency_ms": 35,
    "bitrate_mbps": 80,
    "loss_pct": 0.5,
    "fec_pct": 15,
    "fps": 90,
    "gpu_temp_c": 72,
    "heap_mb": 180,
    "encoder_resets": 0,
    "fovea_active": true
  }
  ```
- 異常検知: 直近5分間でレイテンシが10ms以上増加傾向 → 自動対処（バッファ調整/ビットレート削減）
- ログローテーション: 7日分保持、古いファイルは自動削除

## 将来バージョンとの関係

- **v2.3（USB Type-C PCVR）:** USB Ethernet追加時、GCCの帯域推定はUSBでも動作。ジッタバッファはUSB用のプリセット（5ms固定）を追加予定
- **v2.4（FT強化）:** Foveated Transportの視線データパスを共有。Unified Expressionsのデータ量増加に伴いTCPチャネルの効率化が必要になる可能性
- **v2.5（Companion App拡張）:** セッションログは録画/OBS連携の基盤になる。thermal governorのUI表示もv2.5で追加
