## Overview

v2.2.0でソフトウェア側の実装が完了した。このプレイブックは実機検証から
v2.3リリースまでの全手順を5フェーズで定義する。

```
Phase 0 (30min)    → セットアップ
Phase 1 (2h)       → コア機能検証
Phase 2 (2h)       → v2.2新機能検証
Phase 3 (半日)     → パフォーマンス計測
Phase 4 (1週間)    → 実機依存機能実装 + v2.3リリース
```

各Phaseは前のPhaseの結果に依存する。Phase 1の映像ストリーミングが失敗したら
以降のPhaseには進まない。

> GCC推定器のチューニング複雑性、TCP holdのアーキテクチャ変更、bitrate arbitration

### Step 4.1: GCC推定器 (delay-based帯域推定) — 4日
- [ ] Phase 3の計測データからパケット到着パターンを分析
- [ ] `adaptive/gcc_estimator.rs`: EWMA簡略版実装
- [ ] 合成トレース5種（安定/急激悪化/漸進劣化/ジッタバースト/回復）でテスト
- [ ] TRANSPORT_FEEDBACK (0x12) → GCC推定器に接続
- [ ] **Bitrate arbitration:** GCC = ceiling、loss-based = floor、`clamp(loss_floor, gcc_ceiling)` で調整。Thermal ceilingも統合
- [ ] パラメータ（alpha、閾値）を実機Wi-Fi環境でチューニング（Day 3）
- [ ] テスト追加: delay trend検出、帯域推定精度（±20%以内）、oscillation耐性
- [ ] loss-based controllerとの統合テスト（thermal capあり/なし）
- [ ] 実機で帯域逼迫時のビットレート先行削減を確認
- [ ] config: `gcc_enabled = true`（beta品質のためフラグ付き）

判定: ロス発生前にビットレート削減が開始される

### Step 4.2: TCP holdステートフル化 — 2日
- [ ] `tcp_server.rs`: `listen_and_accept()`を`bind()`（1回）+ `accept_next()`（毎セッション）に分離
- [ ] リスナーをengine.rsのループ外に保持（エンジン全体で1つ）
- [ ] **2トークン設計:** `session_cancel`（ストリーミングタスク用）+ `connection_cancel`（TCP用）。hold中はsession_cancelのみ抑制
- [ ] attemptカウンタをaccept失敗用とconnection-lost用に分離
- [ ] TLS session resumption: hold window（5秒）に限定。hold後はsession ticket無効化
- [ ] hold中のCONFIG_UPDATE拒否（証明書フィンガープリント検証前は制御メッセージ不可）
- [ ] reconnect.rsに抽出（engine.rs複雑度削減）
- [ ] テスト追加（hold中の再接続、タイムアウト、カウンタ分離、session_cancel非発火、audio存続確認）
- [ ] 実機Wi-Fi断テスト: OFF→5s以内ON→自動復帰確認

判定: Wi-Fi断から5秒以内の自動復帰成功、audio途切れなし

### Step 4.3: Thermal Governor — 1日
- [ ] `streaming-engine/Cargo.toml`: nvml-wrapper（常にコンパイル、runtime graceful degrade）
- [ ] `thermal.rs`: NVML API経由でGPU温度5秒ポーリング
- [ ] 温度閾値: 75C警告, 85C制限(解像度90%+ビットレート70%), 90C緊急
- [ ] **BitrateController.max_bitrate_bps に ceiling を設定**（直接bitrate変更ではなく上限設定）
- [ ] 温度低下後30秒かけて品質復帰
- [ ] NVML非対応環境（AMD含む）: runtime init失敗→thermal governor無効化 + ログ警告。パニックしないことを確認
- [ ] GPU抽象化: `trait GpuThermalSource` でNVML実装を分離（AMD対応の将来準備）
- [ ] テスト追加（閾値判定、品質段階、復帰ロジック、init失敗、thermal+adaptive同時発火）
- [ ] 実機で1時間ゲームプレイ中のGPU温度推移を記録

## Hotfix手順（Phase 0-3でバグ発見時）

検証中にバグが見つかった場合:

1. **再現手順を記録** — 何をしたら何が起きたかを正確にメモ
2. **ログ収集** — PC側: streaming-engineのコンソール出力、HMD側: `adb logcat | grep -i fvp > bug_$(date +%s).log`
3. **即時修正可能な場合** — 修正→`cargo test --workspace`→コミット→検証再開
4. **調査が必要な場合** — TODOS.mdにバグとして記録（再現手順+ログパス付き）、検証は次のStepに進む
5. **ブロッカーの場合**（Phase 1 Step 1.1映像が出ない等）— 検証停止、デバッグに専念
